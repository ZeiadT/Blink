Candidate 01

Keep campaign target snapshots out of saved-group catalog

Strong in-process

src/sidepanel/store/campaignStore.ts · src/sidepanel/store/groupStore.ts · src/background/orchestrator.ts · src/background/storage.ts · src/shared/types.ts

Before / one key, two domains

saved lists
Campaign module
run targets
→
→
blink_group_lists
saved interface and campaign implementation share one store

Leakage: campaign target write reappears as saved list.

After / one owner per persistence seam

Saved-group catalog module
normalization + migration inside
Campaign snapshot module
immutable targets inside
→
→
saved-group keys
catalog only
campaign keys
runner only

Two domains; two persistence seams; no mutable target handoff.

Problem

Campaign module writes target groups through saved-group catalog interface; locality breaks when a campaign runs.

Deepen

Campaign snapshot module owns target implementation; saved-group catalog module alone mutates saved-group storage.

Wins

  • • locality: campaigns cannot alter saved groups
  • • leverage: one migration point
  • • interface exposes intent, not transport list
  • • tests hit one persistence seam

Problem

Top recommendation

Separate campaign target snapshots first

One real storage seam currently leaks campaign targets into saved groups. Deepening it restores locality immediately and gives the durable runner, group catalog, and saved-post work stable persistence to build on.
